The audit system is the platform control plane for selective disclosure of private transaction data. It combines a backend service, PostgreSQL storage, scheduled background jobs, and the audit UI.

## Major subsystems

The backend is a NestJS application backed by PostgreSQL. Schema migrations run automatically on startup. Scheduled jobs drive ledger scanning and audit interpretation batches.

| Subsystem | Responsibility |
| --- | --- |
| WorkOS identity | Organizations, users, memberships, OAuth, and Magic Auth email OTP |
| Auth | Maps authenticated sessions to organization and application permission buckets |
| On-chain indexing | Multi-chain ledger scanners (Stellar RPC and Solana indexer clients) that ingest raw audit events |
| Audit storage | Persists encrypted raw audit rows from indexed chain activity |
| Audit interpretation | Decrypts and normalizes indexed events into disclosure-ready records |
| Disclosure and cases | Disclosure requests, investigation cases, transactions, and analytics |
| Applications and contracts | Application registration, contract registry, and org-scoped contract lists |
| Team and organization admin | Organization bootstrap, team sync, and administrator actions |
| Reports | Transaction summary CSV generation and scoped report listing |
| Auditors log | Immutable compliance trail at organization, application, and case scope |
| Audit API | HTTP surface that exposes the subsystems above to the audit UI |
| Audit UI | Organization owner and application workspaces for disclosure workflows |

## Tenant model

Every query is scoped to an organization:

- **Organizations** own team members, applications, and org-level activity
- **Applications** map to contract-backed `foreign_id` values (for example `contract-{id}`)
- **Cases** bind disclosure scope to a specific application and investigation period

An organization owner registers applications and manages team access. Application workspaces expose case review, disclosure requests, reports, and audit logs to users with the appropriate permission keys.

## Audit UI workspaces

The audit UI resolves workspaces from `GET /auth/me`:

| Workspace | Route prefix | Primary users |
| --- | --- | --- |
| Organization owner | `/workspace/organization-owner/*` | Org owners |
| Application | `/workspace/application/:foreignId/*` | App administrators and auditors |

Organization owner navigation includes Overview, Applications, Team, Audit Log, and Reports.

Application workspace navigation includes Overview, Case Review, Disclosure Requests, Reports, Audit Log, and Settings. Nav items are filtered by permission keys.

## End-to-end pipeline

1. **Chain events** — Stellar and Solana transactions emit auditable on-chain activity.
2. **RPC indexers** — On-chain modules scan ledgers and ingest raw events.
3. **Encrypted audit storage** — Indexed rows persist in the audit table as encrypted payloads.
4. **Audit interpretation** — A background runner decrypts and normalizes events into disclosure-ready records.
5. **Interpretation records** — Normalized rows become available for case-scoped queries and analytics.
6. **Disclosure cases** — Approved cases bind users and scope to specific interpretation data.
7. **Audit UI** — Organization owners, administrators, and auditors access cases, transactions, reports, and activity logs through the web application.

See [On-chain indexing](/architecture/on-chain-indexing) for ingestion details and [Disclosure, cases, and reports](/architecture/disclosure-cases-and-reports) for the disclosure lifecycle.
