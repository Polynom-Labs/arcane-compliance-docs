Arcane Compliance is implemented as a Stellar/Soroban privacy-pool contract, a client SDK, a NestJS backend, PostgreSQL, and a React Audit UI. Reference applications are outside the audit platform; they submit Soroban transactions that the audit platform later indexes.

## Component map

| Component | Code location | Responsibility |
| --- | --- | --- |
| Soroban privacy-pool contract | `stellar-privacy-layer-contracts/contract/src/lib.rs` | `PrivacyPoolsContract`, `transact`, storage, Groth16 verification, token transfers, audit event emission |
| Circuits | `stellar-privacy-layer-contracts/circuits` | `Transaction(20,2,2,1,1)`, deposit/withdraw constraints, commitment/nullifier logic, public signals |
| Client SDK | `stellar-privacy-layer-contracts/client-sdk` | `PrivacyPoolSDK`, coin generation, witness preparation, proof generation, stealth addresses, Soroban serialization |
| Backend app | `stellar-privacy-layer-backend/src/app.module.ts` | NestJS module graph, TypeORM, scheduler, chain scanners, auth, cases, reports, logs |
| Stellar scanner | `src/scanner`, `src/stellar-rpc` | Ledger scanning, event parsing, calldata extraction, audit-row upsert, checkpoint update |
| Interpretation worker | `src/audit-interpretation` | Decrypt raw audit rows and write normalized interpretation records |
| API and guards | `src/*/*.controller.ts`, `src/auth`, `src/applications/guards` | Authenticated HTTP routes, permission checks, application route-scope checks |
| PostgreSQL entities | `src/**/entities/*.ts` | Registry, audit, access, disclosure, report, and activity-log tables |
| Audit UI | `stellar-audit-ui/src` | Workspaces, routes, permission-aware navigation, disclosure/case/report screens |

## External applications

Reference or partner applications provide the end-user privacy-pool flow.

Responsibilities:

- Connect to a Stellar wallet.
- Initialize `@auditable/privacy-pool-zk-sdk`.
- Generate or accept a `stpl1` stealth address.
- Build coins, witnesses, Groth16 proofs, public signal bytes, and encoded audit bytes.
- Submit a Soroban `transact` invocation through the user's wallet.
- Show transaction status and application-level errors.

Non-responsibilities:

- Running the Arcane scanner.
- Interpreting audit rows.
- Managing disclosure requests, cases, auditors, reports, or activity logs.
- Enforcing auditor permissions.

## SDK and wallet layer

`PrivacyPoolSDK` is the client-side technical boundary between an application and the Soroban contract.

Important SDK responsibilities:

- `buildStealthAddressSignMessage` and `generateStealthAddressFromStellarSignature` derive the recipient private address from a Stellar wallet signature.
- `generateCoin*` methods construct commitment-compatible coin data.
- `buildWithdrawMerkleWitness` prepares Merkle inclusion data from current pool state.
- `proveWithdrawal` and `proveTransaction` produce proof bytes and public signals for the `Transaction(20,2,2,1,1)` circuit.
- `proofToHex` and `publicToHex` serialize proof data for Soroban.

The wallet signs and submits the Soroban transaction. The Audit UI is not involved in end-user transfer execution.

## Soroban contract

The current contract has a single public transaction entrypoint:

```rust
pub fn transact(
    env: &Env,
    from: Address,
    proof_bytes: Bytes,
    pub_signals_bytes: Bytes,
    encoded: Bytes,
) -> Result<(), Error>
```

`transact` handles the protocol-level deposit, withdrawal, transfer, and mixed transaction shapes through public signals and witness data. These are not separate contract methods.

Contract responsibilities:

- Require authentication from `from`.
- Parse Groth16 proof bytes and public signal bytes.
- Check that `stateRoot` is in recent root history.
- Read nullifier hashes from public signals and reject reused hashes.
- Verify the proof with the stored verification key.
- Store either zero output commitments or exactly two output commitments.
- Store leaf ephemeral BabyJubJub public keys next to commitment leaves.
- Update `LeafCount`, `TREE_ROOT_KEY`, `PairwiseFrontier`, and the 90-slot root-history ring buffer.
- Apply public deposit transfers from `from` to the contract.
- Apply public withdrawal transfers from the contract to the public Stellar account encoded in `withdrawAddressHi` / `withdrawAddressLo`.
- Emit `AuditEncodedDigest` with `message_name = "transact"` and `digest = encoded`.

Public getters expose Merkle root, depth, commitment count, commitments, leaf ephemeral keys, pairwise frontier, nullifier-consumed state, token balance, public slot config, and admin.

## Contract event boundary

The scanner treats the contract event as the on-chain audit boundary.

Current event shape:

```rust
#[contractevent(topics = ["audit", "encoded_digest"], data_format = "single-value")]
pub struct AuditEncodedDigest {
    #[topic]
    message_name: String,
    digest: Bytes,
}
```

The backend maps the `audit` topic to `event_type = transact`. It stores the sealed audit payload as `audit.cyphertext` and stores parsed public signals in `audit.public_signals_json`.

## Application registry

An Arcane application is a backend record, not the same thing as a deployed frontend.

`applications` fields used by the architecture:

- `org_id` scopes the application to one organization.
- `foreign_id` is the stable route/API segment exposed to the Audit UI and application-scoped APIs.
- `application_type` distinguishes `stellar-pool` from `solana-confidential-token`.
- `association.contract_id` links the application to a registered contract row.

The UI route is `/workspace/application/:foreignId`. Application-scoped API routes use `/api/applications/:foreignId/...`. The backend resolves that route segment to an internal `application_id` before permission and query execution.

## Scanner

The Stellar scanner consists of:

- `ScannerModule.register({ stellarChainNames, solanaChainNames })`
- `StellarRpcModule.register({ chainName })`
- `LedgerScannerSchedulerService`
- `StellarLedgerScannerService`
- `performStellarContractScan`
- `buildAuditFromStellarEvent`
- `persistScanResultsQuery`

The scanner lists contracts for a chain, reads ledger ranges from Stellar RPC, filters successful contract events, parses public signal bytes and signer data from `transact` calldata when available, upserts raw audit rows, links them to the chain, and updates `stellar_scans`.

## Interpretation worker

`AuditInterpretationRunnerService` runs every 12 seconds and locks up to 25 uninterpreted audit rows in one database transaction.

For Stellar rows, `interpretStellarAuditRow`:

- Requires `contracts.decoding_key`.
- Parses `audit.public_signals_json`.
- Decrypts and decodes `audit.cyphertext`.
- Plans one or more interpretation rows.
- Replaces existing `audit_interpretation` rows for that audit id.
- Stores a truncated interpretation error on the source `audit` row if processing fails.

## Audit API

The API is the server-side enforcement surface for the Audit UI.

Representative routes:

| Route | Purpose |
| --- | --- |
| `GET /auth/me` | Current organization and permission buckets |
| `GET /api/audit/contract/:contractId` | Contract transaction/audit rows for authorized users |
| `POST /api/applications` | Create an organization application |
| `GET /api/applications/:applicationId` | Read application details by internal id |
| `GET /api/applications/:foreignId/cases` | Application case worklists and case detail routes |
| `POST /api/applications/:foreignId/case-reports` | Case transaction summary report generation |
| `GET /api/applications/:foreignId/reports` | Application report list |
| `GET /api/applications/:foreignId/auditors-log` | Application activity log |
| `GET /api/applications/:foreignId/cases/:caseId/auditors-log` | Case activity log |

Client-side permission checks only shape navigation. API guards remain the source of truth.

## Audit UI

The UI is a React/Vite application with these workspace routes:

| Workspace | Routes |
| --- | --- |
| Organization owner | `/workspace/organization-owner/overview`, `/applications`, `/team`, `/activity`, `/reports` |
| Application | `/workspace/application/:foreignId/overview`, `/cases`, `/cases/new`, `/cases/:caseId`, `/disclosure`, `/reports`, `/log`, `/settings` |

Legacy application auditor/admin routes redirect to `/workspace/application/:foreignId/*`.
