title: "Audit events and interpretation"
description: "Raw audit events, Stellar interpretation, and normalized records used by disclosure workflows."
---

Indexing and interpretation are separate backend stages. Indexing writes raw encrypted `audit` rows from chain events. Interpretation decrypts those rows and writes normalized `audit_interpretation` rows used by cases, reports, and review screens.

## Raw audit events

`audit.event_type` is the scanner-level event type.

| Event type | Source |
| --- | --- |
| `transact` | Current Stellar privacy-pool `AuditEncodedDigest` event and legacy `transact` topic |
| `deposit` | Legacy or compatibility Stellar topic if present |
| `withdraw` | Legacy or compatibility Stellar topic if present |
| `ct_*`, `conf_*`, `public_*` | Solana confidential-token path |

For the current Soroban privacy-pool contract, the public contract method is `transact` and the emitted audit event maps to `event_type = transact`.

## Raw audit schema

| Column | Description |
| --- | --- |
| `id` | Backend audit row id |
| `contract_id` | Registered contract reference |
| `tx_id` | Chain transaction id |
| `soroban_event_id` | Event id from Stellar RPC |
| `event_type` | Scanner-level event type |
| `cyphertext` | Encrypted/sealed audit payload |
| `created_at` | Ledger close time |
| `interpreted` | Interpretation status flag |
| `signer_account` | Transaction signer when resolved |
| `public_signals_json` | Parsed proof public signals |
| `interpretation_error` | Last interpretation error, if any |

The table is unique by `(contract_id, soroban_event_id)`.

## Interpretation runner

`AuditInterpretationRunnerService` runs on a 12-second interval.

Batch behavior:

1. Open a database transaction.
2. Lock up to 25 uninterpreted audit rows.
3. Route Solana confidential-token rows to `interpretCtAuditRow`.
4. Route all other rows to `interpretStellarAuditRow`.
5. Replace interpretation rows for each successfully interpreted audit row.
6. Store a truncated interpretation error on the source audit row if interpretation fails.

The routing step exists because the backend contains both Stellar privacy-pool and Solana confidential-token interpretation code. This page documents the Stellar path.

## Stellar interpretation path

`interpretStellarAuditRow` performs:

| Step | Detail |
| --- | --- |
| Decode-key check | Requires `audit.contract.decoding_key` |
| Public signal parsing | Validates `audit.public_signals_json` |
| Payload decode | Calls `decryptAndDecodeTransaction(cyphertext, decoding_key)` |
| Planning | Calls `planInterpretations(tx, publicSignals)` |
| Persistence | Replaces rows in `audit_interpretation` for `audit_id` |
| Error capture | Writes `audit.interpretation_error` |

Interpretation is backend processing. It does not grant auditor visibility by itself.

## Interpreted records

`audit_interpretation` rows are normalized records.

| Column | Description |
| --- | --- |
| `audit_id` | Source raw audit row |
| `seq` | Sequence when one audit row creates multiple interpreted rows |
| `kind` | Normalized kind: `deposit`, `withdraw`, or `transfer` |
| `payload` | JSONB payload used by case review and reports |
| `amount_stroops` | Parsed amount when available |

The unique key is `(audit_id, seq)`.

## Event type versus interpretation kind

`audit.event_type` and `audit_interpretation.kind` are different levels:

| Level | Example | Meaning |
| --- | --- | --- |
| Raw event | `transact` | The Soroban event/contract-call shape indexed from chain |
| Interpretation kind | `deposit`, `withdraw`, `transfer` | Normalized branch planned from decrypted transaction data and public signals |

A single raw `transact` event can produce multiple interpreted rows.

## Disclosure-aware display

Case and report endpoints query interpreted records only after access checks.

Disclosure flags from `case_requests` control field visibility:

| Flag | Effect |
| --- | --- |
| `full_tx_ids` | Exposes full transaction identifiers |
| `sender_information` | Exposes sender-side fields where available |
| `withdrawal_details` | Exposes recipient or withdrawal-side fields where available |
| `period_from`, `period_to` | Restrict records by approved date range |
| `investigation_contract_addresses` | Restrict records to selected contracts when set |

Fields outside the approved scope remain hidden by API response shaping and UI rendering.

## Error states

Operational states are visible through the source `audit` row:

| State | Indicators |
| --- | --- |
| Not processed | `interpreted = false`, no error |
| Interpreted | Interpretation rows exist for `audit_id` |
| Failed | `interpretation_error` contains the latest error snippet |
| Missing key | Error is written when `contracts.decoding_key` is absent |

Indexing success and interpretation success are separate. A row can be indexed correctly and still fail interpretation.
