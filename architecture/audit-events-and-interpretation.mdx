---
title: "Audit events and interpretation"
description: "Audit event types and the second-phase interpretation pipeline."
---

Audit interpretation transforms encrypted on-chain events into normalized records that disclosure workflows can query. Indexing stores raw events; interpretation decrypts and maps them to a common schema.

## Audit events

An audit event is the encrypted raw record stored after chain ingestion. Each row includes an event type, encrypted payload, and metadata from the indexing phase.

### Stellar privacy-pool events

| Event type | Description |
| --- | --- |
| `deposit` | Tokens locked into the pool and a new commitment added |
| `withdraw` | Commitment spent and tokens withdrawn to a public address |
| `transact` | Combined spend, public withdrawal, and new deposit in one proof |

The exact event payload depends on the privacy-pool contract version. The architecture boundary is stable: the indexer stores raw encrypted rows, and the interpretation worker converts those rows into normalized records.

## Why two phases?

Raw events are chain-specific and encrypted. Disclosure UIs need normalized transaction views with fields such as transaction identifier, period, sender-side information, and recipient-side information.

Interpretation:

1. Reads raw indexed audit rows.
2. Decrypts or resolves encrypted payloads under the configured key model.
3. Validates event structure.
4. Plans one or more interpreted records per audit row.
5. Persists structured payloads for case transaction queries and reports.

Keeping ingestion and interpretation separate allows indexers to run continuously while interpretation catches up in batches.

## Interpreted records

Each interpreted record contains normalized fields used by cases, reports, and review workflows.

| Field | Description |
| --- | --- |
| Source audit row | Link to the raw indexed event |
| Sequence | Position when one audit event produces multiple interpreted records |
| Kind | Normalized operation kind, such as `deposit`, `withdraw`, or `transfer` |
| Payload | Structured transaction fields |
| Amount | Parsed amount when available and permitted |
| Period metadata | Ledger or timestamp fields used for case filtering |

Interpreted records are not automatically visible to auditors. The Audit API still applies organization, application, case assignment, approved disclosure scope, access-window, and permission checks.

## Batch processing

The interpretation worker runs asynchronously.

Typical batch behavior:

1. Open a database transaction.
2. Lock a bounded set of uninterpreted audit rows.
3. Route each row to the Stellar privacy-pool interpretation path.
4. Decrypt or resolve payload data.
5. Validate event structure.
6. Write interpretation rows.
7. Store an interpretation error on the audit row if processing fails.

## Disclosure-aware display

Interpretation payloads feed case transaction tables. Sender, recipient, transaction identifier, and amount fields respect disclosure scope.

Examples:

- A case with transaction identifier scope can expose full transaction identifiers.
- A case with sender information scope can expose sender-side fields where available.
- A case with recipient information scope can expose recipient-side fields where available.
- Fields outside the approved scope remain hidden by policy.

Approved disclosure scope on the case request controls which fields appear in the UI and exported reports.

## Error handling

Interpretation errors are stored with the source audit row so operators can distinguish:

- Events that have not been interpreted yet
- Events that were interpreted successfully
- Events that failed validation
- Events whose encrypted payload could not be decrypted or resolved

Indexing and interpretation failures are separate. A row can be indexed successfully and still fail interpretation.

## Multi-chain extension point

Additional chains or privacy rails require their own raw event adapters and interpretation planners. They can share Arcane's disclosure workflows only after their events are normalized into the same interpreted record boundary.
