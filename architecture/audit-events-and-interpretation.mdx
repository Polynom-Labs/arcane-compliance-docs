title: "Audit events and interpretation"

Audit interpretation transforms encrypted on-chain events into normalized records that disclosure workflows can query. Indexing stores raw events; interpretation decrypts and maps them to a common schema.

## Audit events

An audit event is the encrypted raw record stored after chain ingestion. Each row includes an `event_type`, `cyphertext`, and metadata from the indexing phase.

### Stellar private pool events

| Event type | Description |
| --- | --- |
| `deposit` | Tokens locked; new commitment added |
| `withdraw` | Commitment spent; tokens withdrawn |
| `transact` | Combined spend, public withdrawal, and deposit in one proof |

### Confidential transfer events

| Event type | Description |
| --- | --- |
| `ct_deposit` / `ct_withdraw` / `ct_transfer` | CT token movements |
| `ct_transfer_with_fee` | Transfer with fee component |
| `ct_configure` / `ct_approve_account` | Configuration and approval |
| `conf_mint` / `conf_burn` / `public_mint` / `public_burn` | Mint and burn variants |
| Account lifecycle events | `ct_empty_account`, `close_account`, credit toggles, etc. |

## Why two phases?

Raw events are chain-specific and encrypted. Disclosure UIs need normalized transaction views with sender, recipient, amount, and period filters. Interpretation:

1. Decrypts `cyphertext` using the contract's decoding key
2. Plans one or more interpretation records per audit row
3. Persists structured payloads for case transaction queries and reports

Keeping ingestion and interpretation separate allows indexers to run continuously while interpretation catches up in batches.

## Interpreted records

Each `audit_interpretation` row includes:

| Field | Description |
| --- | --- |
| `audit_id` | Source audit row |
| `seq` | Sequence when multiple interpretations exist per audit |
| `kind` | `deposit`, `withdraw`, or `transfer` |
| `payload` | JSONB with normalized transaction fields |
| `amount_stroops` | Optional parsed amount |

## Batch processing

`AuditInterpretationRunnerService` runs every 12 seconds:

1. Opens a database transaction
2. Locks up to 25 uninterpreted audit rows
3. Routes each row to the Stellar or CT pipeline based on event type
4. Writes interpretation rows or stores an `interpretation_error` on the audit row

Stellar pipeline: decrypt cyphertext via `@auditable/stellar-audit`, run branch planners (deposit/withdraw/transfer), persist results.

CT pipeline: parallel path for Solana confidential transfer payloads.

## Disclosure-aware display

Interpretation payloads feed case transaction tables. Sender and recipient fields respect disclosure scope:

- With `sender_information` enabled, Stellar events resolve sender data from interpretation payloads when available
- Stellar `deposit`/`withdraw`/`transact` rows may show **Hidden by policy** until `signer_account` is mapped from audit metadata
- Recipient is derived via `receiverSideFromPayload` in the cases transaction mapper

Approved disclosure scope on the case request controls which fields appear in the UI and exported reports.
