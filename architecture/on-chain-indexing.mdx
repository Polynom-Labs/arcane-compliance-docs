title: "On-chain indexing"
description: "How the backend scans Stellar RPC, parses Soroban privacy-pool events, and stores raw audit rows."
---

The Stellar indexer reads configured Stellar ledgers through Stellar RPC, filters events for registered Soroban privacy-pool contracts, and persists raw audit rows for interpretation.

The backend also registers Solana scanner providers. This page covers the Stellar/Soroban privacy-pool scanner.

## Runtime components

| Component | Responsibility |
| --- | --- |
| `ScannerModule.register` | Registers scanner providers for configured Stellar and Solana chain names |
| `StellarRpcModule.register` | Loads a `chain` row by name, validates `type = stellar`, creates `@stellar/stellar-sdk/rpc.Server` |
| `LedgerScannerSchedulerService` | Runs bootstrap scan and scheduled Stellar scan ticks |
| `StellarLedgerScannerService` | Lists contracts for one Stellar chain and scans each contract |
| `performStellarContractScan` | Calculates ledger span from checkpoint, RPC retention, and scanner config |
| `collectStellarAuditRows` | Reads ledger pages and collects accepted audit rows |
| `buildAuditFromStellarEvent` | Maps Soroban event and calldata into an `audit` upsert object |
| `persistScanResultsQuery` | Upserts `audit`, writes `audit_chain`, updates `stellar_scans` |

## Scanner scheduling

`LedgerScannerSchedulerService` runs:

- A bootstrap scan at service startup.
- A scheduled Stellar scan using `STELLAR_LEDGER_SCAN_DECORATOR_INTERVAL_MS`.
- A scheduled Solana scan using `SCAN_INTERVAL_MS`.

The Stellar path resolves all configured Stellar scanner chain names from environment configuration, gets their registered scanner services, and calls `scan()`.

Each `StellarLedgerScannerService` prevents overlapping scan cycles for the same service instance. If a previous cycle is still running, the next tick is skipped.

## Contract selection

The scanner does not scan arbitrary chain activity.

For each Stellar chain, it queries registered contracts:

```text
chain(type = stellar) -> contracts(chain_id) -> StellarLedgerScannerService
```

`applications.association.contract_id` links an Arcane application to a registered contract. That link is used later by application-scoped API routes and disclosure workflows.

## Ledger span selection

For each contract, the scanner determines:

| Input | Source |
| --- | --- |
| Previous checkpoint | `stellar_scans.block_end` for `(contract_id, chain_id)` |
| Initial hint | `chain.initial_ledger_hint` or `STELLAR_INITIAL_LEDGER` |
| RPC lower bound | `getLedgers(...).oldestLedger` |
| RPC tip | `getLatestLedger().sequence` |
| Batch size | `SCANNER_LEDGERS_PER_REQUEST` |

The next scan starts at `max(previousEnd + 1, oldestLedger)`. If the next ledger is past the RPC tip, the scan is skipped for that contract.

## Event parsing

For each ledger page:

1. Read ledger metadata from Stellar RPC.
2. Extract Soroban events for the registered contract address.
3. Ignore events that are not in successful contract calls.
4. Read `transact` calldata from ledger metadata when available.
5. Map event topic:

| Event topic | Stored `event_type` |
| --- | --- |
| `audit` | `transact` |
| `transact` | `transact` |
| `deposit` | `deposit` |
| `withdraw` | `withdraw` |

The current contract emits `AuditEncodedDigest` under the `audit` topic with `message_name = "transact"`, so current Stellar privacy-pool rows are stored as `event_type = transact`.

## Raw audit upsert

`buildAuditFromStellarEvent` produces the row upserted into `audit`.

| Column | Value |
| --- | --- |
| `contract_id` | Registered contract id |
| `tx_id` | Stellar transaction hash |
| `soroban_event_id` | Stellar event id |
| `event_type` | Mapped event type |
| `cyphertext` | Sealed audit bytes from the event value |
| `created_at` | Ledger close timestamp |
| `interpreted` | `false` |
| `signer_account` | Signer resolved from calldata or `unknown` |
| `public_signals_json` | Parsed public signals from event payload or calldata |
| `interpretation_error` | Cleared on ingest |

The `audit` table has a unique key on `(contract_id, soroban_event_id)`. Rescanning the same ledger range updates the same row instead of duplicating it.

After the audit upsert, the scanner inserts into `audit_chain` with `ON CONFLICT DO NOTHING`.

## Checkpoint update

The same database transaction updates `stellar_scans`:

| Column | Meaning |
| --- | --- |
| `contract_id` | Contract being scanned |
| `chain_id` | Stellar chain being scanned |
| `block_start` | Start of the persisted scan span |
| `block_end` | End of the persisted scan span |
| `last_signature` | Reserved cursor field |

The unique key is `(contract_id, chain_id)`, so each contract/chain pair has one checkpoint row.

## Failure behavior

The scanner catches and logs per-contract scan errors. A failure for one contract does not stop the scheduler from scanning other contracts in the same tick.

Operationally relevant failure categories:

| Failure | Effect |
| --- | --- |
| No contracts for chain | Scanner logs and returns |
| RPC retention moved past checkpoint | Start ledger clamps to `oldestLedger` |
| RPC/network failure | Scan cycle logs an error; checkpoint is not advanced for failed work |
| Parser returns no audit row | Event is ignored |
| Duplicate event | Upsert updates the existing `audit` row |
| Interpretation failure | Stored later on `audit.interpretation_error`; indexing and interpretation failures are separate |
